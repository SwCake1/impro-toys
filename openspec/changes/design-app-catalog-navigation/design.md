## Контекст

Репозиторий представляет собой статический сайт без зависимостей. Корневой `index.html` сейчас содержит заголовок и текстовый список из одного пункта, а `tools/bell/index.html` — самостоятельный полноэкранный интерфейс с большим объёмом локальных стилей и без способа вернуться на главную. Будущие инструменты также будут размещаться на независимых страницах внутри `tools/`.

Каталог должен масштабироваться при появлении новых инструментов, работать на настольных и мобильных устройствах и ощущаться живым благодаря анимациям, не превращаясь в визуально перегруженный интерфейс. Навигация должна сочетаться с ярко выраженным индивидуальным оформлением инструментов. В проекте нет системы сборки, компонентного фреймворка или клиентского роутера.

## Цели и не-цели

**Цели:**

- Превратить корневую страницу в понятный анимированный каталог доступных инструментов.
- Гарантировать две колонки на телефонах и увеличивать их количество при наличии места на широком экране.
- Дать каждому отдельному инструменту единый прямой путь обратно в каталог.
- Сформировать небольшие переиспользуемые соглашения по оформлению для будущих инструментов.
- Сохранить относительные URL, статическое размещение и адаптивность в браузерах.

**Не-цели:**

- Создание карточек для ещё не реализованных инструментов.
- Реализация будущих пасхалок; дизайн лишь оставляет для них пространство.
- Унификация внутреннего визуального оформления всех инструментов.
- Создание бесшовного одностраничного роутинга или подключение фреймворка переходов.
- Добавление отдельного варианта с уменьшенным количеством анимации в рамках этого изменения.

## Решения

### Использовать сетку карточек на основе нативных ссылок

Каждая карточка будет одной семантической ссылкой, а не контейнером с отдельной кнопкой запуска. Благодаря этому вся карточка становится кликабельной, а работа с клавиатурой, URL и поведение без JavaScript предоставляются браузером.

Рассмотренная альтернатива: JavaScript-обработчики нажатия на универсальных контейнерах. Отклонена, поскольку они дублируют нативную навигацию и требуют дополнительной реализации доступности.

### Использовать CSS Grid с двумя мобильными колонками и гибкой широкой раскладкой

На телефонах сетка будет явно состоять из двух равных колонок. Начиная с ширины, на которой карточки комфортно помещаются, сетка будет автоматически подбирать количество колонок по доступному пространству вместо набора брейкпоинтов для конкретных устройств. Центрированный контейнер с максимальной шириной не даст контенту чрезмерно растянуться на очень больших экранах.

Рассмотренная альтернатива: две фиксированные колонки на всех экранах. Отклонена как неэффективная для настольных экранов. Фиксированные брейкпоинты на две, три и четыре колонки также отклонены как менее гибкие по сравнению со встроенным подбором размеров.

### Сохранить спокойную базовую композицию и выражать характер через взаимодействие

Страница будет использовать сдержанные типографику, отступы, цвета и простую согласованную графику. Более выразительные движения будут связаны прежде всего с наведением, фокусом, нажатием и касанием. Анимации могут быть заметными и игровыми, но подписи и основные действия останутся стабильными и читаемыми.

Для анимаций каталога по возможности будут использоваться CSS-трансформации, прозрачность и псевдоэлементы. Небольшие JavaScript-улучшения допустимы для эффектов, зависящих от положения указателя, или будущих пасхалок, однако основная навигация никогда не будет зависеть от них.

Рассмотренная альтернатива: насыщенная иллюстрациями театральная тема. Отклонена, поскольку постоянный декор противоречит требованию о визуально чистом интерфейсе.

### Использовать собственную графику проекта

В качестве идентификаторов инструментов будут использоваться единообразные встроенные или общие SVG-изображения вместо зависящих от платформы эмодзи. Это сохранит цельность визуального языка без подключения библиотеки иконок.

Рассмотренная альтернатива: эмодзи. Отклонена, поскольку их вид меняется между платформами и делает каталог визуально непоследовательным.

### Добавить на страницы инструментов прямые ссылки в каталог

Каждый инструмент будет содержать понятно подписанную ссылку, которая напрямую ведёт в корневой каталог. История браузера использоваться не будет, поскольку инструмент может быть открыт из закладки или по общей ссылке. Элемент получит общий структурный стиль, который конкретный инструмент сможет адаптировать через CSS-переменные или локальные переопределения.

Поскольку страницы находятся по адресам `tools/<name>/`, ссылка на каталог будет относительной к этому уровню вложенности. Пути к общим ресурсам будут строиться аналогично.

Рассмотренная альтернатива: общий JavaScript, динамически вставляющий навигацию. Отклонена для первоначальной реализации, потому что небольшое повторение ссылки надёжнее, работает до загрузки скриптов и соответствует разрешённому архитектурой дублированию простой разметки.

## Риски и компромиссы

- [В двух мобильных колонках мало места для текста] → Использовать короткие подписи, близкие к квадрату карточки и не показывать длинные описания на узких экранах.
- [Насыщенные анимации могут сдвигать интерфейс или мешать навигации] → Анимировать визуальные свойства и декоративные слои, а не размеры карточек или расположение текста.
- [На сенсорных устройствах нет состояния наведения] → Предусмотреть явную реакцию на касание и сохранить понятность назначения карточки до взаимодействия.
- [Полноэкранное оформление инструмента может конфликтовать с общей навигацией] → Разместить её в свободном верхнем углу и разрешить ограниченную настройку темы или положения при неизменных подписи и поведении.
- [Каталог и инструменты могут рассинхронизироваться при добавлении новых страниц] → Включить добавление карточки и элемента возврата в описанную процедуру создания каждого будущего инструмента.

## План перехода

1. Добавить только необходимые каталогу и навигации общие ресурсы.
2. Заменить текущий список на корневой странице новым каталогом, сначала содержащим только существующий звонок.
3. Добавить переход в каталог на страницу звонка и убедиться, что он не конфликтует с основной кнопкой и настройками.
4. Проверить навигацию и адаптивность через прямые локальные URL на типичных размерах телефона и настольного экрана.

Для отката достаточно восстановить прежнюю корневую страницу и удалить добавленный элемент возврата и общие ресурсы. Изменения постоянных данных или миграция URL не требуются.

## Открытые вопросы

Точные палитру, типографику, графику и характер движений можно выбрать во время реализации в рамках чистого, слегка театрального направления и заданных требований к интерактивности.
